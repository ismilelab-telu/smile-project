A good ML project often wins or loses before the model is trained. The reason is simple: the team needs to agree on **what should be predicted**, **when the prediction is made**, and **what data may be used**.

Formulating the problem means turning a real need into something a model can work on. We do not stop at "make estimates more accurate". We make it specific.

## Steps to Formulate a Problem

Use this order:

1. **Define the real goal.** For example, a food delivery team wants to give more realistic arrival estimates.
2. **Turn the goal into an ML question.** For example, "how many minutes will an order likely take to reach the customer?"
3. **Define the model output.** The output can be a number, category, score, recommendation, or group.
4. **Define the prediction time.** For example, when the order is placed, when the courier is assigned, or after certain information is available.
5. **Define the available features.** Features must already be known when the model makes a prediction.
6. **Define the problem type.** Numeric output points to regression, category output points to classification, and groups without a target point to clustering.
7. **Define the evaluation.** The team needs a way to judge whether the model actually helps.

## Target and Features

In supervised learning, the **target** is the value or category we want to predict. Examples include delivery time, tomorrow's sales, or whether a customer will churn.

**Features** are the information used by the model as input. Features must be available when the prediction is made. If information is only known after the event ends, it is unsafe because it can leak the answer.

For food delivery:

- business goal: give more realistic delivery-time estimates;
- ML problem statement: predict `Delivery_Time_min` for an order;
- target: delivery time in minutes;
- safe features: `Distance_km`, `Weather`, `Traffic_Level`, `Time_of_Day`, `Vehicle_Type`, `Preparation_Time_min`, and `Courier_Experience_yrs`, as long as those values are available when the estimate is made;
- unsafe feature: `Delivery_Time_min` reused as a model input;
- problem type: regression, because the target is a number.

## Metadata Is Not the Main Feature

Some columns help us read the dataset, but are not always useful as model inputs. `Order_ID` is a common example.

Columns like this are usually **metadata**:

- they help identify rows;
- they are useful for audit or tracing;
- they do not directly explain delivery duration.

## Why This Matters

A clear problem statement keeps the model from aiming at the wrong thing. Without a clear target, the model does not know what to learn. Without a clear prediction time, features can get mixed with future information.

In this lesson, you will use food delivery data to distinguish **target**, **safe features**, and **metadata**. Assume the displayed safe features are already available when the estimate is made.
