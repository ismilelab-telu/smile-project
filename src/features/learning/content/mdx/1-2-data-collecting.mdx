Data collection is not just grabbing files. It connects a real question with records that can answer it. If the question is vague, the dataset usually becomes vague too.

Before collecting columns, define what the model should help decide. A demand forecast, a churn model, and a risk detector need different rows, fields, time windows, and labels.

## Start from the Question

A useful collection step begins with a concrete question. For example, "predict drink demand before the next shift starts" is clearer than "analyze cafe data."

The clearer question tells the team what each row should represent, what target might be needed, and what information must be known before prediction time.

## Track Source and Timing

Good datasets have context. You should know where each source came from, when it was captured, how often it updates, and whether it describes the same event as the target.

Timing is especially important in ML. A column may be valid for reporting but unsafe for prediction if it is only known after the target happens.

## Check Permission and Purpose

Data should be usable for the intended purpose. That includes privacy, consent, business rules, and any restrictions around sensitive fields.

Collecting every possible column first and deciding the purpose later is risky. It creates noise, increases privacy exposure, and can lead to features that are not appropriate for the model.

## Useful Collection Notes

Record:

- where each source came from;
- when the data was captured;
- what each row represents;
- what each field means;
- whether the field is available at prediction time;
- whether the data can be used for the intended purpose;
- which important fields are missing or uncertain.

Good data collection makes later modeling easier to trust. Bad collection forces the team to guess what the dataset means.
